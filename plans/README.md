## Findings considered and rejected

### From previous audit (2026-06-26, commit 75746d9)

- **CORRECTNESS-03** (unhandled backup failure crashes): Not planned separately because A6's path traversal fix + B3's round-trip test will tighten backup error handling as a side effect.
- **CORRECTNESS-05** (injectVitePlugin hardcoded named-import special case): Low impact (only affects future plugins), not worth the disruption to a shared patching utility. Plan only if a new named-import Vite plugin is added.
- **CORRECTNESS-08** (resolveStatusesWithTiming mutable array): Timing data is diagnostic-only and unused; fixing the array pattern adds noise without user benefit.
- **SECURITY-02** (PID in temp filenames): Low-risk information leak; fixing would add a crypto import for negligible gain.
- **PERF-04** (sequential stat calls in detectBundler): ~24 stat calls locally, impact ~2-5ms; not worth optimizing.
- **PERF-03** (no cache TTL): File-change-based invalidation already covers most cases; a TTL safety net adds complexity for a rare edge case. C3 covers the substantive cache issue.
- **DX-03/04/05/06**: Low impact DX nits — not worth diverting from higher-leverage work.
- **DOCS-01 through DOCS-05**: Documentation improvements are valuable but lower priority than the correctness and testing gaps above.
- **DIR-01** (agent catalog standalone): Folded into D2 which is more actionable.

### From current audit (2026-06-30, commit 3cc6838)

- **Off-by-one diff header (diff.ts:67-69)**: Line count inflation is display-only. The actual patch content is correct. Fix is trivial but impact is LOW. Re-addressing if someone adds downstream consumers of the diff header metadata.
- **__proto__ via backupFile (backup.ts:58-60)**: Requires `filepath` to be `"__proto__"`, which no task currently produces. Defensive fix is cheap but impact is LOW. Revisit if the backup index is extended to user-configurable paths.
- **npx --yes -y (skills-install.ts:223-228)**: Supply-chain mitigation. Impact is LOW because the agent-catalog values are hardcoded. Not worth adding CI complexity for.
- **patchJson O(n*m) re-parse (json-merge.ts:100-122)**: Each `modify()` call re-parses the document. For typical config files (~30-70 lines with 5-10 patches), the impact is negligible. Worth fixing if users report slow apply on large JSON files.
- **Plugin dynamic import validation (plugins.ts:92)**: Requires write access to project config. The plugin system is still being designed (per plan 019/D3). Add validation as part of the full plugin API stabilization.
- **Duplicated deepEqual**: Package boundary (ADR-002) prevents sharing. Both implementations are correct. Not worth creating a shared package for.
- **Module-level pkg cache (task.ts:27-38)**: Currently safe. Only a risk if someone refactors the apply pipeline to interleave check/apply. Flag for review during any apply refactor.
- **Effect v4 beta risk**: Documented risk, no action needed now. Revisit when Effect v4 reaches stable.
- **Effect usage inconsistency (fs.ts vs backup.ts vs apply.ts)**: Architectural preference, not a bug. The code works correctly under all three patterns. Standardizing would be churn for zero user benefit.
- **Inquiry module over-engineered**: The 7-file NLP pipeline was deliberately designed per ADR-024. The "no AI" constraint was explicit. Revisit if task catalog grows beyond 50+ tasks and search quality degrades.
- **Workspace package discovery (DIR-02)**: Deliberately deferred in ADR-023. Design spike needed first — not an implementation plan.
- **Plugin system docs (DIR-01 remainder)**: Will be addressed when plan 019 is fully executed and the API stabilizes.
